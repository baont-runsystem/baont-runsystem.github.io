Mở đầu

Tài liệu này giải thích các khái niệm quan trọng về Version Control System (VCS), cách dùng Git (hệ thống quản lý phiên bản phân tán — DVCS) và GitHub (dịch vụ lưu trữ remote repository và nền tảng hợp tác).

Hiểu ý nghĩa của Version Control System

  • Ghi lại lịch sử thay đổi mã nguồn theo commit.
  • Tương tác nhiều lập trình viên cùng lúc, cho phép so sánh, quay lại, gộp thay đổi.
  • Giúp theo dõi ai làm gì, khi nào, và tại sao (khi viết commit message tốt).

Khởi tạo repository rỗng

Để tạo một repository Git mới trong thư mục hiện tại:

git init

Giải thích: lệnh git init tạo thư mục ẩn .git chứa metadata và object database. Sau init, thư mục trở thành một Git repository (local).

Xóa repository

Nếu muốn xóa repository (local) — xóa thư mục .git. Cẩn thận: hành động này sẽ mất lịch sử.

# linux / mac
rm -rf .git

# windows (PowerShell)
Remove-Item -Recurse -Force .git

Các lệnh Git cơ bản

git status

Hiển thị trạng thái working directory và staging area.

git status

git add

Đưa thay đổi vào staging area (index) trước khi commit.

# thêm 1 file
git add file.txt
# thêm tất cả thay đổi (files + modifications)
git add -A
# thêm chỉ modifications và new files (không xóa):
git add .
# interactive
git add -p

Một số flag hay dùng: -p (patch, chọn hunks), -A, --intent-to-add.

git commit

Tạo một commit (bắt snapshot từ staging area).

git commit -m "Tiêu đề ngắn"
# nếu chưa add, commit tất cả thay đổi tracked files
git commit -a -m "Sửa nào đó"
# edit message interactively
git commit

Thường dùng: -m, -a, --amend (sửa commit mới nhất - chỉ khi chưa push hoặc biết hệ quả).

git log

Xem lịch sử commit.

git log
# xem dạng 1 dòng
git log --oneline
# xem graph
git log --oneline --graph --all --decorate

Hiểu stages khi add & commit

Ba trạng thái chính:

  1. Working directory — nơi bạn sửa file.
  2. Staging area (index) — nơi bạn dùng git add để chuẩn bị commit.
  3. Repository (HEAD) — nơi commit được lưu (history).

Cấu hình Git

Git có các cấu hình cấp hệ thống, cấp user (global), và cấp repository (local).

Thay đổi cấu hình global

git config --global user.name "Tên Bạn"
git config --global user.email "you@example.com"
# hiển thị
git config --global --list

Tạo alias

Alias tạm thời (chỉ terminal session):

# bash
alias gst='git status'
# powershell
Set-Alias gst git status

Alias cố định (git alias):

git config --global alias.st status
# dùng: git st
# ví dụ khác
git config --global alias.co checkout
# dùng: git co

Tổng quan GitHub

GitHub là gì — so sánh với Git

Git là một hệ thống VCS (công cụ). GitHub là một dịch vụ hosting remote repositories, cung cấp giao diện web, issue tracker, actions, PR workflows và nhiều tính năng cộng tác.

Tại sao dùng GitHub? So sánh với DVCS khác

  • Tích hợp mạng xã hội (PR, review, CI/CD).
  • Hỗ trợ public/private repo, pages, actions.
  • Alternatives: GitLab, Bitbucket — đều là nền tảng tương tự; lựa chọn phụ thuộc vào tính năng CI, hosting, policy doanh nghiệp.

(Lưu ý: phần khởi tạo repository rỗng và xóa đã trình bày ở phần mở đầu - vì cả Git local và repository remote có các bước tương ứng.)

Làm việc với Remotes & SSH

Tạo remote repository trên GitHub

Trên GitHub: nhấn New repository, đặt tên, chọn public/private, rồi tạo.

Tạo remote repository bằng CLI

Sử dụng GitHub CLI (gh) — ví dụ:

# tạo remote repo (yêu cầu đã login gh auth login)
gh repo create ten-repo --public --source=. --remote=origin --push

Push code từ local lên remote

# thêm remote
git remote add origin git@github.com:username/ten-repo.git
# push nhánh chính lần đầu kèm tạo upstream
git push -u origin main
# push các lần sau
git push

SSH keys — ý nghĩa và tạo

SSH key dùng để xác thực an toàn đến remote (không cần nhập mật khẩu mỗi lần). Thường tạo 1 cặp: private (cần giữ bí mật) và public (đưa lên GitHub).

# tạo SSH key (rsa hoặc ed25519)
ssh-keygen -t ed25519 -C "you@example.com"
# hiển thị public key
cat ~/.ssh/id_ed25519.pub
# copy lên GitHub > Settings > SSH and GPG keys

Sau khi add key, kiểm tra:

ssh -T git@github.com

GitHub workflow cơ bản

  1. Tạo fork (nếu bạn không có quyền push trực tiếp): click Fork trên GitHub — tạo bản sao trong account của bạn.
  2. Clone remote repository: git clone git@github.com:username/ten-repo.git
  3. Tạo branch riêng (feature branch): git checkout -b feature/my-change
  4. Commit và push branch: git push -u origin feature/my-change
  5. Trên GitHub: tạo Pull Request từ branch của bạn vào upstream repository.

Upstream — thiết lập

# giả sử origin là fork của bạn
git remote -v
# add upstream (original repo)
git remote add upstream git@github.com:org/original-repo.git
# fetch và rebase/sync
git fetch upstream
git merge upstream/main
# hoặc
git rebase upstream/main

Check out và Reset

git checkout, git clean, git rm --cached

# chuyển branch hoặc checkout file
git checkout main
git checkout -- file.txt   # lấy lại file từ HEAD

# xóa file không tracked/ignored (cẩn thận)
git clean -f    # chỉ files
git clean -fd   # files + directories

# xóa khỏi staging area nhưng giữ file
git rm --cached file.txt

Undo commit với git reset

Reset thay đổi pointer của HEAD; có 3 chế độ:

  • --soft: chỉ di chuyển HEAD (staged giữ nguyên).
  • --mixed (mặc định): di chuyển HEAD và reset index (staging area), giữ working directory.
  • --hard: di chuyển HEAD, reset index và working directory — mọi thay đổi sẽ mất (không thể khôi phục dễ dàng).
# ví dụ
git reset --soft HEAD~1
git reset --mixed HEAD~1
git reset --hard HEAD~1

Branching

Tạo, liệt kê, xóa branch

# tạo và chuyển
git checkout -b feature/x
# liệt kê
git branch
# xóa local
git branch -d feature/x
# xóa remote
git push origin --delete feature/x

Ý nghĩa branch & khi dùng

Branch cho phép phát triển tính năng độc lập, thử nghiệm, sửa lỗi mà không ảnh hưởng code ổn định trên main hoặc release.

Merge branch

# merge feature vào main
git checkout main
git merge feature/x
# nếu muốn fast-forward
git merge --ff-only feature/x
# hoặc rebase trước khi merge để lịch sử sạch
git checkout feature/x
git rebase main
git checkout main
git merge feature/x

Xem sự khác nhau của commit

Sử dụng git diff để xem thay đổi giữa working dir, staging, và commit:

# thay đổi chưa staged so với HEAD
git diff
# thay đổi staged so với HEAD
git diff --staged
# xem diff giữa 2 commit
git diff commit1..commit2
# xem commit detail (patch)
git show COMMIT_HASH

Có nhiều tools GUI để visualize diff: GitKraken, SourceTree, GitHub Desktop, VSCode Git extension (rất tiện).

Merge Conflicts

Hai loại merge chính: fast-forward (di chuyển pointer) và recursive (3-way merge, có thể tạo conflict).

Conflict xảy ra khi Git không thể tự động kết hợp thay đổi; cần can thiệp thủ công:

  1. Mở file báo conflict — Git chèn các marker <<<<<<<...
  2. Chỉnh sửa để chọn/ghép nội dung đúng.
  3. git add file và git commit để hoàn tất merge.

Kiến thức nâng cao

Stashing

Sử dụng khi bạn cần tạm thời cất thay đổi chưa commit để chuyển branch:

git stash          # stash changes
git stash list
git stash apply     # áp dụng stash nhưng vẫn giữ stash
git stash pop       # áp dụng và xoá stash

Rebasing

Rebase thay đổi lên một base mới, giúp lịch sử tuyến tính hơn.

# rebase branch lên main
git checkout feature/x
git rebase main

So sánh Rebase vs Merge: Rebase tạo lịch sử sạch (không có merge commits) nhưng thay đổi commit ids (không an toàn với commit đã public). Merge giữ lịch sử đầy đủ và an toàn khi nhiều người làm việc trên cùng branch.

Interactive rebase

git rebase -i HEAD~5
# trong interactive mode bạn có thể: pick, reword, edit, squash, fixup, drop
# dùng để sửa message, gộp commit, sắp xếp lại commit

Reverting

git revert tạo một commit đảo ngược một commit trước đó (an toàn cho lịch sử public). git reset di chuyển HEAD và có thể rewrite lịch sử (không an toàn nếu đã push).

# tạo commit đảo ngược
git revert COMMIT_HASH
# reset (ví dụ)
git reset --hard COMMIT_HASH
